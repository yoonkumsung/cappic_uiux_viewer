# CAPP!C 프로젝트 규칙

## 용어 규칙

* **전문 용어 대신 쉬운 용어로 풀어서 사용한다.** IT 비전문가(축구교실 코치, 풋살장 운영자)가 바로 이해할 수 있는 쉬운 말로 써야 한다.
* 아래 매핑을 반드시 따른다:
* 전문성이 필요한 디버그 화면 등을 제외하고 UI나 버튼에 표시되는 모든 텍스트는 초등학생도 이해할 수 있는 수준으로 작성한다.
* 기술적 수치(중첩 퍼센트 등)는 사용자에게 한번에 노출하지 않는다(토글은 가눙). 상태만 보여준다 (정상/오류).

## 작업 규칙

* **지시한 사항은 하나도 빠짐없이 반영한다.** 여러 항목을 지시받으면 하나하나 차근차근 전부 처리한다.
* 빠뜨린 항목이 없는지 완료 후 반드시 체크한다.
* HTML 수정 시 div 태그 균형(open/close 개수)을 반드시 확인한다.
* 수정 후 기존 화면이 깨지지 않았는지 직접 스크린샷을 찍어서 검증한다.
* **사용자가 요청한 내용을 절대 무시하지 않는다.** 여러 요청이 들어오면 하나도 빠짐없이 모두 처리한다. 나중에 하겠다고 미루지 않는다.
* **게으르게 답변하지 않는다.** scale=1.0 같은 하드코딩, 임시방편 금지. 반응형으로 근본 해결한다.
* **코드 수정 후 사용자에게 제공하기 전에 반드시 bug\_history.md를 읽고 기존 버그가 재발하지 않는지 검증한다.** (memory/bug\_history.md 참고)
* 검증 항목: JS 문법, 변수 선언 순서, 요소 참조 유효성, 여러 해상도 시뮬레이션, transform/layout 충돌, iframe 로딩 타이밍

## 기능별 검증 규칙 (Phase 구현 시 필수)

* **기능 하나 구현할 때마다 아래 절차를 반드시 수행한다. 건너뛰기 금지.**

### 1. 단위 검증
* 구현한 기능 자체가 정상 동작하는지 직접 스크린샷을 찍어서 확인한다.
* 스크린샷은 냉정하게 검증한다. "될 것이다"가 아니라 "직접 눈으로 봤다"로 판단한다.
* 텍스트가 잘리거나, 레이아웃이 밀리거나, 색상이 빠졌거나 하는 미세한 문제도 잡아낸다.

### 2. 연계 안정성 검증
* 수정한 파일과 연관된 모든 기능을 점검한다:
  - index.html 수정 → overview.html의 iframe 로딩, HTML 추출(ziExtractHTML), generate-thumbs.js 정상 동작 확인
  - styles.css 수정 → index.html의 75개 화면 + overview.html 양쪽에서 스타일 깨짐 없는지 확인
  - JS 수정 → go() 함수, data-go 네비게이션, 팝업/모달, 토글 등 기존 인터랙션 정상 동작 확인
* 수정하지 않은 화면이 깨지지 않았는지 최소 5개 화면을 샘플링하여 확인한다:
  - 필수 샘플: adm-01, con-04, con-feed, editor, web-02 (각 카테고리 대표)

### 3. 회귀 방지
* 기존 bug_history.md의 12개 버그가 재발하지 않았는지 체크한다.
* 특히 주의: canvas zoom 페이지 압축, iframe 로그인 화면 고정, cardHeights 초기화 오류, embed 모드 CSS 미적용
* overview.html의 connections[]과 실제 data-go 네비게이션이 일치하는지 확인한다.

### 4. 완료 보고 형식
* 각 기능 구현 완료 시 아래 형식으로 보고한다:
  - 구현 내용 (무엇을 했는지)
  - 단위 검증 결과 (스크린샷 기반)
  - 연계 검증 결과 (영향받는 기능 목록 + 확인 결과)
  - 회귀 확인 (bug_history 대조 결과)
